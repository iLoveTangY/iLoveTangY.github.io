服务器项目总结

发表于 2019-08-05 | 评论数： | 阅读次数：

服务器项目

介绍项目

背景：去年开始准备找工作时写了一些总结性的文章，然后为了能够随时都能复习，在Github上基于Hexo搭建了个人博客，但是由于服务器在境外，访问速度比较慢。后面阿里云做活动，学生买云服务器有优惠，就买了一台云服务器，想着在云服务器上搭建个人博客。一开始用的Ngnix，用的也挺好。后面学了网络编程后就想着自己写一个服务器，然后就做了这个项目。

技术要点：

  • 采用非阻塞I/O，Reactor模式，基于Epoll边沿触发的I/O多路复用；
  • 使用线程池避免线程频繁创建销毁的开销；
  • 使用基于小根堆的定时器关闭超时连接；
  • 使用双缓冲技术实现了简单的异步日志系统；
  • 为了减少资源泄漏的可能，使用智能指针等RAII机制；
  • 使用eventfd来唤醒睡眠线程。

Q&A

定时器是怎么实现的？还有什么实现方式？

每一个事件循环中有一个定时器，用来管理本事件循环需要处理的那些Http连接，定时器直接采用STL里的优先队列来实现，优先队列中的元素是每个Http请求的超时时间。但是由于优先队列不支持随机访问，当一个连接超时时并不会直接从优先队列中弹出，而是采用类似于Lazy Delete的策略，只有这个节点前面的所有节点都被删除后这个节点才会被真正删除。

还可以采用timing wheel（时间轮盘），可以采用循环队列来实现。循环队列的大小就是默认超时时间，循环队列的尾指针指向的元素就是即将超时的连接，每一秒都会向后移动尾指针的位置，然后断开当前位置的连接。当一个连接收到数据后就将其移到尾指针所指的位置，增加它的寿命。

双缓冲异步日志是什么？为什么这样做？压力测试？

一般来说，在多线程程序中实现线程安全的日志库并不难，简单的方法就是用一个全局的互斥量保护对日志文件的I/O，但是这种情况下会造成全部线程争抢一个锁，并且由于文件读写速度较慢，会严重影响程序的效率；另外也可以每个线程单独写一个日志文件，但是这样的话主线程有可能会阻塞在磁盘的读写上，并且多个日志文件也不好进行分析。

基于以上的原因，我这里采用了异步日志，也就说，用一个背景线程负责收集日志消息，并写入日志文件，其他的业务线程只管往这个日志线程发送日志消息。所谓双缓冲就是准备两个缓冲区A和B，前段负责往A填写日志消息，后端负责将B中的数据写入文件。当A写满之后，交换A和B，前段写B，后端将A的数据写入文件，如此往复。这样的好处是前端线程在新建日志消息的时候不必等待磁盘操作。实际实现采用了四个缓冲区，这样可以进一步减少前段日志的等待。

Epoll和Select/Poll的区别？

Select是通过在内核中检查存放描述符标志位的数据结构来判断是不是有事件发生的。Select处理的是描述符数组，它能监听的描述符数量有限，一般为1024个，这个系统调用只有一个接口，当有大量描述符需要监听时需要将所有的这些描述符拷贝到内核空间，如果这些描述符中有事件发生，再将所有的这些描述符拷贝回用户空间，再让用户遍历这些返回的描述符来判断哪个描述符上发生了事件。这样每次调用Select都需要做一来一回两次大量描述符的拷贝，效率非常低。Poll本质上来说和Select没有任何区别，只是去除了监听的描述符数量的限制。

Epoll不同，它将调用Select中添加描述符和等待事件发生拆分为两个系统调用：epoll_wait和epoll_ctl。这样当你需要监听大量描述符时只需要在最开始的时候将描述符添加到epoll的等待队列中，后面就不需要再做这种大量描述符的拷贝工作了。epoll_wait返回的也是就绪的描述符，而不是所有需要监听的描述符，也就是说从内核空间拷贝到用户空间的也只是少量的就绪描述符，不像Select一样需要将所有的描述符都拷贝回来再一一判断。而且epoll_wait返回后要重新监听也不需要将所有的描述符重新拷贝回内核空间。

另外Epoll还有LT和ET两种触发模式，能够根据需求采取不同的模式。

Epoll水平触发和边沿触发的区别？

Level Triggered(水平触发, LT)：当被监控的文件描述符上有可读写事件发生时，epoll_wait()会通知处理程序去读写。如果这次没有把数据一次性全部读写完(如读写缓冲区太小)，那么下次调用 epoll_wait()时，它还会通知你在上没读写完的文件描述符上继续读写，当然如果你一直不去读写，它会一直通知你。

Edge Triggered(边缘触发, ET)：当被监控的文件描述符上有可读写事件发生时，epoll_wait()会通知处理程序去读写。如果这次没有把数据全部读写完(如读写缓冲区太小)，那么下次调用epoll_wait()时，它不会通知你，也就是它只会通知你一次，直到该文件描述符上出现第二次可读写事件才会通知你。

Epoll在实现上是采用红黑树加就绪链表的设计。当调用epoll_ctl时实际上是往红黑树中插入了一个节点，当有事件发生时内核会调用一个回调函数，将发生事件的描述符添加到就绪链表中去。如果这之后我们调用epoll_wait，会对就绪链表中的描述符再进行筛选，然后把准备好的的描述符拷贝到用户态空间中，在这之后，如果采用的是LT模式，会将这些就绪的描述符再次放入就绪链表，也就是说下次调用epoll_wait()时这些描述符仍然还在就绪链表中，这就是LT模式在一个描述符上如果有数据没有读完下次调用epoll_wait()还会再次通知的原因。

在我的项目中使用的是边缘触发搭配非阻塞I/O。

一个请求到来的具体处理过程？

首先，服务器启动时会在主线程，也就是主事件循环中将listenfd添加到epoll的监听队列中。当一个Http请求到来时，epoll_wait()会返回，然后在事件循环中会进行事件处理，进而异步回调注册的处理函数，在这里，是一个处理连接的成员函数。在这个函数里面，会调用accept来建立连接，从事件循环线程池中获取一个线程，也就是一个事件循环，然后将建立的连接进行封装后分配给获取的线程，最后会将这个连接加入该这个线程的epoll的监听队列中。

然后如果一个已经建立连接的acceptfd可读，同样也会在该连接所属线程的epoll_wait中返回，进行事件处理，进而回调注册的处理函数，比如可读时的处理，可写时的处理等等。

线程的唤醒方式？什么是eventfd？优点是啥？

一般来说，本项目中的线程都是睡眠在epoll的等待队列上的，如果要唤醒这样的一个线程，只需要该线程处理的描述符中有事件发生即可。所有可以像epoll中注册一个描述符，当你想唤醒该线程时只需要往该描述符中写入数据即可。

在我的这个项目中采用的是eventfd来唤醒一个线程。eventfd 是 Linux 的一个系统调用，专门创建一个文件描述符用于事件通知。eventfd主要是用来做多进程/线程间的通信的，我这里只是用来做线程内部通信。eventfd的主要优点是节省内存并且效率高，主要是因为创建一个eventfd并不是真正创建或者打开了一个文件，只是在内核中创建了一个数据结构，内核的维护成本比较低。

如何检查内存泄漏？

我在Linux下使用的是Valgrind，它的MemoryCheck工具能够报告比较详细的内存使用信息。比如对内存的越界访问，malloc/free，new/delete不匹配等等。为了减少内存泄漏，程序中也使用了智能指针等RAII技术。

什么是Reactor模式？

同步的事件循环+non-blocking I/O。

为什么用非阻塞I/O？

因为服务器的实现中采用了进程池的机制，如果使用阻塞I/O，当一个线程被Read给阻塞之后，这个线程就什么都不能干了。如果线程池中总共有n个线程，来了第n+1个长连接，那么还是没法处理。这样就没法做到高并发了。

用到了哪些锁？这些锁的区别是什么？

项目中主要用到了互斥锁和条件变量。

互斥锁是一种简单的加锁的方法来控制对共享资源的访问，互斥锁只有两种状态,即上锁(lock)和解锁(unlock)。如果一个线程锁定了一个互斥量，在它解除锁定之前，没有其他线程可以锁定这个互斥量。如果一个线程已经锁定了一个互斥量，第二个线程又试图去锁定这个互斥量，则第二个线程将被挂起（不占用任何cpu资源），直到第一个线程解除对这个互斥量的锁定为止，第二个线程则被唤醒并继续执行，同时锁定这个互斥量。

条件变量用来等待某件事情发生，主要包括两个动作：一个线程等待”条件变量的条件成立”而挂起；另一个线程使”条件成立”（给出条件成立信号）。为了防止竞争，条件变量的使用总是和一个互斥锁结合在一起。

如何解决死锁？

死锁的出现有四个必要条件：

  • 互斥使用
  • 不可抢占：资源只能自愿放弃
  • 请求和保持：进程必须占有资源，再去申请
  • 循环等待：在资源分配图中存在一个环路

理论上来说，解决死锁有很多办法，比如死锁预防：可以一次性申请所有需要的资源，但是它需要你能预知未来，还可以对资源排序，资源申请必须按序进行，但是这种方法会造成资源的利用率低；再还有死锁避免的银行家算法：就是判断此次的资源请求是否会引起死锁，但是每次资源申请时都执行这个算法效率太低了。现在一般来说个人的PC操作系统都是采用的死锁忽略的方式。
