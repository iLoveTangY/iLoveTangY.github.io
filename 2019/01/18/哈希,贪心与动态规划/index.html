哈希函数与哈希表

哈希函数与哈希表

在一般的存储结构（线性表、树等）中，元素在结构中的位置是相对随机的，因此如果要查找某个元素都需要进行一系列和元素的比较，差的时间复杂度在$O(N)$，好的时间复杂度也在$O(logN)$级别。而理想情况下我们想要$O(1)$的时间复杂度，也就是说我们希望得到一个元素能够通过某种计算直接得到其位置，而不是通过比较来找到位置。哈希函数就是这样一种计算方法，能够在存储位置和存储元素之间建立一个确定的对应关系，使每个元素都能和一个唯一的存储位置相对应。

哈希表中元素是由哈希函数确定的。将数据元素的关键字K作为自变量，通过一定的函数关系（哈希函数），计算出的值，即为该元素的存储地址。表示为：$Addr=H(key)$。

为此在建立一个哈希表之前需要解决两个主要问题：

  1. 均匀性

    $H(key)$的值均匀分布在哈希表中；如果关键字的计算结果在存储空间上的分布不均匀，那么就会产生大量的浪费空间，增加哈希表的时空复杂度。

    MD5 和 SHA1 可以说是目前应用最广泛的Hash算法，而它们都是以MD4为基础设计的。它们都能很好的保证均匀性。

  2. 冲突的处理

    冲突：在哈希表中，不同的关键字值对应到同一个存储位置的现象。即关键字$K1≠K2$，但$H(K_1)= H(K_2)$。均匀的哈希函数可以减少冲突，但不能避免冲突。发生冲突后，必须解决；也即必须寻找下一个可用地址。

    一般来说解决冲突的办法很多，比如拉链法、开放定址法、桶定址法……目前应用最多的还是拉链法。即一个位置后面串接一条链表，如果产生冲突，则把冲突的元素依次放在链表中。如果觉得在链表中查找太慢了，还可以一个位置后面接一棵红黑树，加快查找速度。

设计RandomPool结构

请设计这样一种RandomPool结构，支持以下三种操作：

  • delete(key)：将key从结构中移除；

布隆过滤器

如果想要判断一个元素是不是在一个集合里，一般想到的是将所有元素保存起来，然后通过比较确定。链表，树等等数据结构都是这种思路. 但是随着集合中元素的增加，我们需要的存储空间越来越大，检索速度也越来越慢($O(N)$,$O(logN)$)。不过我们还学了哈希表，知道它查找元素的时间复杂度为$O(1)$；比如如果我们做一个浏览器，有一百亿个64字节长的URL的黑名单，每次检索到一个URL时都需要将其和黑名单中的URL作比较，如果使用哈希表的话我们可以将所有的URL存到一个哈希表里面，每次判断检索到的URL在哈希表中是否存在，这样时间复杂度为$O(1)$。但是这里面存在一个问题：空间占用太多。在这道题里面如果把所有URL都存储起来占用空间为640GB，如果URL更多的话占用的空间也就更多。为了解决这个问题，布隆于1970年提出了布隆过滤器。

布隆过滤器使用$K$个哈希函数和$M$个比特的位向量。初始时为了将黑名单URL存储起来，将URL通过这$K$个哈希函数计算后的结果模上$M$，得到一个$0$~$M-1$之间的数，再将位向量上相应的位置设置为1。对每个URL都进行这个操作，最后再查找的时候同样将得到的URL通过这$K$个哈希函数，如果位向量上对应的每个位置值都为1，那么就说明这个URL在黑名单上。很明显，这个方法是有一定缺陷的，比如如果我们的$M$选的太小，而要URL太多，那么可能添加完所有的URL之后整个位向量的所有值都为1了，这样后面查询的时候所有的URL都在黑名单里面了。但是只要我们$K$和$M$取得比较合适，那么是可以将出错的概率降低到给定值以下的。下面给出布隆过滤器中的计算公式：

一致性哈希

使用普通Hash的问题

假如我们有大量的数据，一台服务器放不下的数据，还有4台服务器。为了利用起来这些服务器，我们决定将每条数据经过hash函数的运算后放置在相应的服务器上，比如采用hash(data) % 4这个函数。这样在查找这些数据时只需要经过同样的哈希运算就能找到存放数据的服务器。

但是这种普通的hash方式存在问题：假如我们新加入了一台服务器或者有一台服务器失效了，那么所有的数据都需要重新进行哈希！我们应该想办法不让这种情况发生，但是由于上述Hash算法本身的缘故，使用取模法进行缓存时，这种情况是无法避免的，为了解决这些问题，Hash一致性算法（一致性Hash算法）诞生了！

一致性哈希的神秘面纱

一致性Hash算法也是使用取模的方法，只是，刚才描述的取模法是对服务器的数量进行取模，而一致性Hash算法是对$2^{32}$取模，什么意思呢？简单来说，一致性Hash算法将整个哈希值空间组织成一个虚拟的圆环，如假设某哈希函数H的值空间为$0$~$2^{32}-1$（即哈希值是一个32位无符号整形），整个哈希环如下：

整个空间按顺时针方向组织，圆环的正上方的点代表0，0点右侧的第一个点代表1，以此类推，2、3、4、5、6……直到$2^{32}-1$，也就是说0点左侧的第一个点代表$2^{32}-1$， 0和$2^{32}-1$在零点中方向重合，我们把这个由$2^{32}$个点组成的圆环称为Hash环。

并查集

并查集的数据结构

在计算机科学中，并查集是一种树型的数据结构，用于处理一些不交集（Disjoint Sets）的合并及查询问题。有一个合并-查找算法（union-find algorithm）定义了两个用于此数据结构的操作：

  • Find：确定元素属于哪一个子集。它可以被用来确定两个元素是否属于同一子集。
  • Union：将两个子集合并成同一个集合。

以上内容来自维基百科。通俗来说，并查集这个数据结构就是用来合并集合以及判断某个元素是否在集合里的。因此，并查集这个数据结构只定义了两种操作：find和union，具体功能如上所述。

贪心策略

贪心算法，是一种在每一步选择中都采取在当前状态下最好或最优（即最有利）的选择，从而希望导致结果是最好或最优的算法。

递归与动态规划

暴力递归

  1. 把问题转化为规模缩小了的同类问题的子问题
  2. 有明确的不需要继续进行递归的条件(base case)
  3. 有当得到了子问题的结果之后的决策过程
  4. 不记录每一个子问题的解（区别于动态规划）

  1. 从暴力递归中来
  2. 将每一个子问题的解记录下来， 避免重复计算
  3. 把暴力递归的过程， 抽象成了状态表达
  4. 并且存在化简状态表达， 使其更加简洁的可能

可以看到，有很多的重复计算，比如$f(1,1)$。这样递归版本的时间复杂度就很高。当然，为了降低复杂度，你可以将已经计算过的值保存下来，比如将$f(1,1)$保存到一个map中，后续用到它的时候直接查就好，这就是记忆化搜索的方式。这里我们介绍动态规划的方式。

并不是所有有重复计算的递归都能改成动态规划。一个递归方式若要能改成动态规划，还需满足一个条件——无后效性。所谓无后效性，是指如果在某个阶段上过程的状态已知，则从此阶段以后过程的发展变化仅与此阶段的状态有关，而与过程在此阶段以前的阶段所经历过的状态无关。具体到这道题来说就是$f(1,1)$的值和到达$f(1,1)$的路径无关：$f(1,1)$的值只和后面的值有关，而和是通过$f(0,1)$到达$f(1,1)$还是通过$f(1,0)$到达$f(1,1)$无关。在满足了无后效性的条件之后从递归改成动态规划就是一个高度套路化的过程：

  1. 观察递归版本，看有几个变量就生成一个几维的dp表。这道题中有两个变量i和j（matrix是固定的），并且取值范围就是数组的维度，那么我们应该生成一个二维的dp表。
  2. 观察BaseCase，在dp表中填充所有BaseCase所在的位置。这道题中的BaseCase就是当i为最后一行或者j为最后一列时。
  3. 根据BaseCase填充表中的其他位置。
